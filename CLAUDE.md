# 编程学习工具

辅助开发者进行编程学习的软件

## 编码规范

- Python 遵循: `~/.claude/rules/python` 规范
- TypeScript 遵循: `~/.claude/rules/typescript` 规范

## 开发规范

### 思考先行

**先想清楚再动手，不隐藏疑惑，不凭空假设。**

- 不确定的事情明确说出来，不猜、不编造
- 多种解读并存时，列出选项而非默认选一个
- 有更简单方案时直接指出，该拒绝就拒绝
- 遇到模糊点先停下来搞清楚，再继续

### 简单至上

**用最少的代码解决问题，不做投机性设计。**

- 不写需求之外的功能
- 不为单次调用造抽象
- 不预设"灵活性"或"可配置性"
- 不处理不可能发生的错误场景
- 200 行能写清楚的不写 500 行
- 三段相似代码好过一次不必要的抽象

### 手术式修改

**只动需要动的地方，只清理自己制造的垃圾。**

- 不改相邻代码的格式、注释、风格
- 不顺手重构不相干的东西
- 匹配已有代码风格，即使和你的偏好不同
- 你引入的 import / 变量 / 函数如果不再使用，删除干净
- 发现与当前任务无关的死代码，提出来但不擅自删除

### 目标驱动

**把模糊需求转化为可验证目标，循环直到确认通过。**

- "加个校验" → 先写无效入参的测试，让它通过
- "修这个 bug" → 先写复现测试，让它通过
- "重构 X" → 确保重构前后所有测试通过
- 多步任务先列步骤，每步说清验证标准

### 研发流程

- 遵循 `skill: karpathy-guidelines` 开发准则
- 方案设计、需求开发、bug 修复遵循 superpower 流程
- 编写代码采用 TDD 范式（先写测试，再写实现）
- UI 设计采用 `skill: frontend-design`
- 代码检索优先使用 `codegraph` 系列工具
- 编写后必须检视、评审、修改，直到满足规范

## 注意事项

1. **所有用户可见文案必须走 i18n**：前端界面中任何用户可见的文本（包括 placeholder、hint、tooltip、message 提示等）都必须通过 `t('key')` 从 `translations.ts` 获取，禁止硬编码中/英文。新增字段时同步补充 `zh` 和 `en` 两个字典的 key。

2. **禁止提交没有验收的内容**：功能未经用户确认验收，禁止 commit。

3. **禁止魔鬼数字**：阈值、超时、并发数等常量统一在 `config.py` 中管理，代码中不得出现裸数字。

4. **禁止猜测**：对不确定、未验证的事情，必须先做详细分析、验证，再给结论。

5. **必须保证测试通过**：BUG 修复和需求开发的 UT 测试、E2E 测试必须全部通过。

6. **前端 E2E 验证**：采用 `skill:playwright-cli` 进行浏览器端验证。
