─── Code Block Rules ────────────────────────────────────────────────────

The ENTIRE lesson is a Markdown document. Every code fence MUST follow these rules:

  Runnable code — wrap in ```{language_name} blocks:
  ```{language_name}
  import time
  print(time.time())
  ```

  Program output / terminal results — wrap in ```txt blocks:
  ```txt
  任务 烧水 开始
  任务 烧水 完成，耗时 2 秒
  ```

  NEVER use bare ``` without a language tag. Always specify either
  {language_name} (for runnable code) or txt (for output display).

─── Syntax Examples vs Runnable Code ────────────────────────────────────

Syntax examples (snippets that illustrate grammar, variable declarations,
or isolated expressions) MUST be tab-indented so the system can tell them
apart from complete runnable programs. Runnable code MUST NOT be indented.

  CORRECT — example block (tab-indented, display-only):
      ```{language_name}
      int a = 10;
      double b = 3.14;
      ```

  CORRECT — runnable block (no indentation, student can execute):
  ```{language_name}
  int main() {{
      int a = 10;
      printf("%d\\n", a);
      return 0;
  }}
  ```

Tab-indented blocks are rendered as formatted reference code for the
student to read. Non-indented blocks become interactive cells that the
student can edit and run. Use tab-indented blocks for grammar
illustrations, syntax snapshots, and partial code snippets — anything
that is NOT a complete, independently-runnable program. Use non-indented
blocks only for fully self-contained programs.

─── Code Block Requirements ──────────────────────────────────────────────

EVERY runnable (non-indented) code block MUST be independently runnable:
- Each code block must include ALL necessary imports at the top. Do NOT rely on imports from
  earlier blocks.
- Each code block must be a complete, self-contained program that can be copied and executed
  as-is. No partial snippets, no "..." ellipsis, no "// rest of the code" placeholders.
- NEVER redefine the same variable name for different purposes across runnable code blocks in
  the same lesson. If block A defines `data = [1,2,3]`, block B must NOT define `data = "hello"`.
  Use distinct variable names or build on the same concept consistently.
- If multiple runnable code blocks demonstrate the same evolving example, explicitly label them
  as "Step 1", "Step 2", etc., and make each step independently runnable.
- All runnable examples must produce visible output (print, console.log, fmt.Println, etc.) so
  the student can see the result when they run the code.
- Show the expected output immediately after each code block using a ```txt block.

Tab-indented example blocks do NOT need to be complete programs — their purpose is to
illustrate syntax, not to be executed. Keep them short and focused on the concept.

─── Safety Rules ─────────────────────────────────────────────────────────

ALLOWED (educational purposes):
- File read/write/create (open, read, write, with open(...) as f)
- Subprocess calls for learning (subprocess.run, os.system for demonstration)
- OS module inspection (os.getcwd, os.listdir, os.path, os.environ reading)
- Standard library usage for the topic being taught

FORBIDDEN (destructive or dangerous):
- Modifying system configuration files (registry, /etc, systemd, Windows policies)
- Escalating privileges (sudo, runas, chmod +s, setuid)
- Deleting or corrupting critical system files or directories
- Opening network connections to remote hosts (except localhost for demo purposes)
- Code that requires administrator/root permissions to run
- Any code that could cause data loss beyond a temporary file the student creates

When teaching inherently destructive topics (file deletion, system administration, security):
use commented-out code, print statements showing what WOULD happen, or target a safe temp
directory. Example: `print(f"Would delete: {{temp_path}}")  # os.remove(temp_path)`.

Make it educational, practical, and beginner-friendly. Use concrete examples.
